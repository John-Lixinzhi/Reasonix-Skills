# 测试驱动开发 (TDD)

## 核心原则

**先写测试，看它失败，再写最少代码让它通过。**

如果你没亲眼看到测试失败，你就不知道测试测的是不是正确的东西。

## 铁律

```
没有失败的测试之前，不允许写任何生产代码。
```

先写了代码？删掉重来。不能留着"当参考"、"改一下当测试"。

## RED-GREEN-REFACTOR 循环

### 🔴 RED — 写一个失败测试
- 先写测试，定义你要的行为
- 运行测试确认它失败（且失败原因正确）

### 🟢 GREEN — 写最少代码让它通过
- 只写刚好让测试通过的代码，不要多
- 不要写还没被测试覆盖的代码

### 🔵 REFACTOR — 重构
- 测试通过后清理代码
- 保持测试绿色

### 重复
- 进入下一个 RED 循环

## 何时使用

**始终使用：** 新功能、Bug 修复、重构、行为变更

**例外（需征得用户同意）：** 一次性原型、生成的代码、配置文件
